# **Section 188.** 회복 / 병행제어
---

## 회복 ( Recovery )

트랜젝션을 수행하는 도중 장애가 발생해서 DB가 손상됐을 때 정상 상태로 복구하는 작업

**장애 유형**
- `트랜잭션 장애` : 입력 데이터 오류, 불명확한 데이터, 시스템 자원 요구의 과다 > `트랜잭션 내부의 비정상적인 상황`으로 인해 프로그램 실행 중지
- `시스템 장애` : DB 손상은 아니지만 하드웨어 오동작, 소프트웨어 손상, 교착 상태 등 `트랜잭션의 연속적인 수행`에 장애
- `미디어 장애` : 저장장치인 디스크 블록의 손상이나 디스크 헤드의 충돌에 의해 DB의 일부 또는 전부가 `물리적 손상된 상태`


**회복 관리기** ( Recovery Management )
- DBMS의 구성요소 중 하나
- 트랜잭션 실행이 성공적으로 완료되지 못하면 트랜잭션이 DB에 생성했던 변화들을 취소시키고 수행 이전의 상태로 **복구**하는 역할
- 메모리 덤프, 로그 를 이용해 회복 수행


## 회복 기법 종류

### 연기 갱신 기법 ( Deferred Update )

- 트랜잭션이 성공적 완료가 될 때까지 실질적인 갱신을 연기
- 트랜잭션 수행 중 갱신되는 내용은 **log**에 저장
- 부분완료 시점에 log 내용을 실제 DB에 기록
- 부분완료 전 장애 발생 시 Rollback
- Redo 작업만 가능
> `Redo (재시도)` : 덤프와 로그를 이용해 가장 최근의 정상적인 DB로 회복 시켜 트랜잭션을 재실행 시킴


### 즉각 갱신 기법 ( Immediate Update )

- 트랜잭션이 데이터 갱신하면 부분 완료 전이라도 DB에 반영
- 회복 작업을 대비해서 갱신된 내용은 Log에 보관
- Redo와 Undo 모두 가능


### 그림자 페이지 대체 기법 ( Shadow Paging )

- 갱신 이전 DB를 페이지 단위로 구성 후 복사본 보관 > 그림자 페이지
- 장애 발생 시 그림자 페이지로 대체하여 회복 시킴
- 로그와 Undo 및 Redo 알고리즘이 필요 없음

### 검사점 기법 ( Check Point )

- 트랜잭션의 특정 단계에서 재실행할 수 있도록 갱신 내용, 상황에 관한 정보 등 검사점을 로그에 보관
- 회복 시 전체 철회가 아닌 검사점부터 회복하여 회복 시간 절약

### 한 줄 정리

- **연기 갱신 기법** > Log에 저장하며 부분완료 시 DB 갱신, Redo만 가능
- **즉각 갱신 기법** > 트랜잭션을 DB에 즉시 반영하면서 Log에 갱신 내용 보관, Redo, Undo 가능
- **그림자 페이지 대체 기법** > DB를 복사하여 보관하다가 회복 시 복사본으로 대체, Redo, Undo, Log 필요없음
- **검사점 기법** > 검사점을 두어 회복 시 검사점 부터 회복

## 병행 제어 ( Concurrency Control )

동시에 여러 **트랜잭션 병행 수행할 때** DB의 일관성을 파괴하지 않도록 **트랜잭션 간의 상호 작용 제어**를 하는 것


### 병행 제어 목적

- 데이터베이스의 공유를 최대화
- 시스템의 활용을 최대화
- 데이터베이스의 일관성 유지
- 사용자 응답 시간 최소화


## 병행 제어 기법 종류

### 로킹 ( Locking )

- 주요 데이터의 액세스를 `상호 베타적`으로 함
> `상호 베타적` : 같은 카테고리에 있지만 서로 다른 상태
- `로킹 단위`를 액세스 하기 전에  Lock을 요청해서 Lock이 허락되어야 로킹 단위를 액세스할 수 있도록 하는 기법
**로킹 단위 ( Locking Granularity )** : 
- 한번에 로킹할 수 있는 객체의 크기이다. 
- DB, file, record, field 등이 단위가 될 수 있음
- 크면 로크 수가 작아 관리가 쉽지만 병행성 수준이 낮아짐
- 작으면 로크 수가 많아 관리가 복잡해 오버헤드가 증가하지만 `병행성 수준`은 높아짐
> `병행성 수준` : 병행성 수준이 낮으면 DB 공유도가 감소한다는 의미 높으면 공유도 증가


### 타임 스탬프 순서 ( Time Stamp Ordering )

- 직렬성 순서 결정을 위해 트랜잭션 간 처리 순서를 미리 선택 > 가장 보편적임
- 트랜잭션 실행 시작 전에 Time Stamp에 기록 > 이에 따라 작업 수행
- 교착 상태 발생 안함


### 최적 병행수행 ( 검증 기법, 확인 기법, 낙관적 기법 )

- 트랜잭션이 Read Only 트랜잭션이면 충돌률이 매우 낮아서 일관성 유지한다는 점을 이용한 기법


### 다중 버전 기법

- 타임 스탬프의 개념을 이용한 기법; 다중 버전 타임 스탬프 기법이라고도 함
- **타임 스탬프** > 시간표로 관리; **다중 버전 기법** > 버전 부여하며 관리


### 한 줄 요약

- **로킹** : 로킹 단위로 액세스, 상호 베타적
- **타임 스탬프 순서** : 타임 스탬프 사용, 교착 상태 발생 안함
- **최적 병행 수행** : Read Only
- **다중 버전 기법** : 버전 부여하며 관리


## 병행 수행의 문제점

|문제점|의미|
|---|------|
|갱신 분실 (Lost Update)|두 개 이상의 트랜잭션이 같은 자료 공유하여 갱신할 때 갱신 결과의 일부가 없어지는 현상|
|비완료 의존성 (Uncommitted Dependency)|- 하나의 트랜잭션 수행이 실패한 후 회복되기 전에 다른 트랜잭션이 실패한 갱신 결과를 참조하는 현상<br>- 임시 갱신이라고도 함|
|모순성 (Inconsistency)|- 두 개의 트랜잭션이 병행 수행될 때 원치 않는 자료 이용해서 발생하는 문제<br>- 불일치 분석이라고도 함|
|연쇄 복귀 (Cascading Rollback)|병행수행되던 트랜잭션들 중 하나에 문제가 생겨서 Rollback하면 다른 트랜잭션도 같이 Rollback함|
