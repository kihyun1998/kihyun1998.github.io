# **Section 191.** Secure SDLC
---

## Secure SDLC란?

보안상 안전한 소프트웨어를 개발하기 위해 `SDLC`에 보안 강화를 위한 프로세스를 포함한 것
> `SDLC` : 소프트웨어를 개발하기 위한 모든 과정; Software Development Lift Cycle

- 유지 보수 단계에서 보안 이슈를 해결하기 위해 소모되는 많은 비용을 최소화하기 위해 등장
- 요구사항 분석, 설계, 구현, 테스트, 유지보수 등 전체 단계에 걸쳐 수행되어야할 보안 활동 제시

**대표적인 방법론**

- `CLASP` : 
    - Secure Software 사에서 개발
    - 초기 단계에서 보안 강화를 위해 개발
    - 활동 중심, 역할 기반의 프로세스로 구성, 현재 운용중인 시스템에 적용하기 적합
- `SDL` :
    - 마이크로소프트에서 SDLC를 개선
    - 전통적인 나선형 모델을 기반
- `Seven Touchpoints` :
    - 보안 모범사례를 SDLC에 통합
    - 설계 및 개발 과정의 모든 산출물에 대해 위험 분석 및 테스트
    - SDLC의 각 단계에 관련된 7개 보안 강화 활동


## 보안 활동

### 요구사항 분석 단계에서의 보안 활동

보안 항목에 해당하는 요구사항을 식별하는 작업

- 정보들을 보안 요소별로 등급을 구분하여 수준별로 분류
- 정보보호 보안 정책을 참고하여 항목 출처, 요구 수준, 세부 내용 문서화


### 설계 단계에서의 보안 활동

식별된 보안 요구사항들을 소프트웨어 설계서에 반영 및 보안 설계서 작성

- 위협에 대해 보안대책, 예산, 영향 범위, 대응책 수립
- 보안 통제 기준을 수립하고 설계에 반영
- **네트워크** : 네트워크 분리 및 방화벽 설치
- **서버** : 보안이 뛰어난 운영체제 사용, 보안 업데이트, 외부 접속 접근 통제
- **물리적 보안** : 출입통제, 개발 공간 제한, 폐쇄회로, 감시 설비
- **개발 프로그램** : 허가되지 않은 프로그램 통제, 데이터 무결성 검사


### 구현 단계에서의 보안 활동

표준 코딩 정의서 및 소프트웨어 개발 보안 가이드 준수하며 설계서에 따라 보안 요구사항들 구현

- 지속적인 단위 테스트 > 보안 취약점 최소화
- 코드 점검 및 소스코드 진단 작업 > 소스코드 안정성 확보


### 테스트 단계에서의 보안 활동

설계 단계에서 작성한 보안 설계서를 바탕으로 보안 사항들이 정확히 반영되고 동작하는지 점검

- 동적 분석 도구 또는 모의 침투를 통해 위협 해결여부 검증
- 구현 단계에서 추가로 제시된 위협들 점검
- 모든 결과 문서화로 보존, 개발자에게 피드백


### 유지보수 단계에서의 보안 활동

이전 과정을 모두 수행해도 발생할 수 있는 보안 사고 식별 및 해결하고 보안 패치

## 보안 요소

3대 요소로는 기밀성, 무결성, 가용성이 있고 그 외에 추가로 인증, 부인 방지가 있다.

- `기밀성` : 인가된 사용자에게만 접근 허용, 노출된 데이터 읽을 수 없음
- `무결성` : 인가된 사용자만 수정 가능
- `가용성` : 인가된 사용자는 언제든 사용 가능
- `인증` : 사용자가 합법적인 사용자인지 확인하는 행위 ex) 패스워드, 지문
- `부인 방지` : 데이터 송수신한 자가 부인할 수 없도록 증거 제공
