# **Section 25.** 모듈
---

## 모듈

- 모듈은 다른 모듈에서 접근이 가능해야 함.
- 단독 컴파일이 가능하고 재사용할 수 있음.
- `기능적 독립성` : 모듈의 기능이 서로 독립됨, 하나의 기능만 수행하고 다른 모듈과 과도한 상호작용을 하지 않음
- 독립성이 높으면 모듈 수정하여도 다른 모듈들에게 거의 영향을 미치지 않음, 오류 쉽게 발견
- 결합도가 약하고 응집도가 강하고 모듈의 크기가 작으면 독립성이 높음

## 결합도 ( Coupling )

모듈 간에 상호 의존하는 정도, 두 모듈 사이의 연관 관계
어떤 모듈이 다른 모듈을 의존하는 정도

### 특징

- 결합도가 약할수록 품질 높음
- 결합도가 강하면 시스템 구현 및 유지보수가 어려움
- 결합도의 종류는 자료 결합도, 스탬프 결합도, 제어 결합도, 외부 결합도, 공통 결합도, 내용 결합도가 있다.
- 자료 결합도가 가장 결합도가 낮고 내용 결합도가 가장 결합도 높음

### 자료 결합도 ( Data Coupling )

- 모듈 간 인터페이스가 자료 요소로만 구성될 때 결합도
- 모듈들이 파라미터를 통해 데이터를 공유하는 경우
- 한 모듈의 내용을 변경하더라도 다른데 영향 없음

### 스탬프(검인) 결합도 ( Stamp Coupling )

- 모듈 간 인터페이스로 배열이나 레코드 등의 자료구조가 전달될 때 결합도
- 두 모듈이 동일한 자료 구조를 조회하는 경우의 결합도
- 데이터 구조를 공유
- 실제로 조회하지 않는 모듈에게도 영향을 미침

### 제어 결합도 ( Control Coupling )

- 논리적 흐름 제어를 위해 제어 요소를 전달하는 결합도
- 처리 기능이 두 모듈에 분리되어 있는 경우, 한 모듈을 통제하는 경우 발생
- 하위 > 상위 모듈로 제어 신호 이동 ( 권리 전도 현상 발생 )

### 외부 결합도 (External Coupling )

- 어떤 모듈에 있는 데이터를 외부의 다른 모델이 참조할 때 결합도
- 참조하는 데이터 범위를 모듈에서 제어할 수 있음

### 공통(공유) 결합도 ( Common Coupling )

- 공유되는 데이터를 여러 모듈이 사용할 때 결합도
- 공통 데이터를 수정하면 사용하는 모듈에 영향을 미치기에 모듈 독립성을 약하게 만듬

### 내용 결합도 ( Content Coupling )

- 한 모듈이 다른 모듈의 내부 기능, 자료를 참조, 수정할 때 결합도
- 내부로 제어가 이동한ㄴ 경우에도 내용 결합도


## 응집도 ( Conhesion )

정보 은닉 개념의 확장, 모듈의 내부 요소드의 서로 관련되어 있는 정도

### 특징

- 관련 있는 것 끼리 응집되어 있으면 응집도가 강하고 반대는 약함.
- 응집도 강할 수록 품질 높음

### 기능적 응집도 ( Functional Cohesion )

내부의 모든 기능 요소들이 단일 문제와 연관되어 수행될 경우

### 순차적 응집도 ( Sequential Cohesion )

하나의 활동으로부터 나온 출력 데이터를 그 다음 활동 입력 데이터로 사용하는 경우

### 교환(통신)적 응집도 ( Communication Cohesion )

동일한 입출력을 사용해서 다른 기능을 수행하는 구성 요소들이 모였을 경우

### 절차적 응집도 ( Procedural Cohesion )

다수의 관련 기능을 갖을 때 구성 요소들이 기능을 순차적으로 수행할 경우

### 시간적 응집도 ( Temporal Cohesion )

특정 시간에 처리되는 기능을 모아 하나의 모듈로 작성한 경우

### 논리적 응집도 ( Logical Cohesion)

유사한 성격, 특정 형태로 분류되는 처리 요소로 형성된 경우

### 우연적 응집도 ( Coincidental Cohesion )

서로 관련 없는 요소로만 구성된 경우



## 팬인 ( Fan-In ) / 팬아웃 ( Fan-Out )

- 팬인은 모듈에 들어가는 수
- 팬 아웃은 모듈에서 나가는 수
- 시스템의 복잡도를 알 수 있음
- 팬인이 높으면 재사용 측면에서 설계 잘됨, 그러나 단일 장애점이 발생할 수 있어서 관리 및 테스트 필요
- 팬아웃이 높으면 불필요한 모듈 호출 검토해보고 단순화 시킬 수 있는지 검토
- 팬인 높고 팬아웃 낮으면 시스템 복잡도 최적화


## N-S 차트

논리의 기술에 중점을 둔 도형을 이용한 표현 방법

### 특징

- 제어 논리 구조 표현
- GOTO, 화살표 사용안함
- 조건 복잡한 곳 명확히 식별하는데 적합
- 선택, 반복 구조를 시각적 표현
- 이해 쉽고 코드 변환 용이, but 작성 어려움
- 총체적 구조 표현과 인터페이스 나타내기 어려움
- 단일 출입구로 표현
