## 요구사항 검토 방법

- `동료 검토` : 작성자가 명세서 내용 설명, 동료들이 결함 발견하는 형태

- `워크 스루` : 검토 회의 전 명세서를 미리 배포하여 사전 검토해서 짧은 검토 회의를 통해 오류 조기 검출

- `인스펙션` : 명세서 작성자를 제외한 다른 검토 전문가들이 확인하면서 결함 발견

## 기능적 요구 vs 비기능적 요구

- **기능적 요구사항** : 시스템이 실제 어떻게 동작하는지에 관점
ex) 시스템이 뭘 하는지, 어떤 기능?, 입출력 무엇, 어떤걸 저장, 반드시 수행해야하는 기능, 제공 받기 원하는 기능
- **비기능적 요구사항** : 시스템 구축에 대한 `성능`, `보안`, `품질`, `안정` 등 실제 수행에 보조적인 요구사항 ( like 속도 등..)
    **성능** : 처리 속도, 처리량 등 
    **보안** : 시스템의 데이터 및 기능, 운영 접근 통제
    **품질** : 품질 평가 대상에 대한

### UML 다이어그램 - 관계

- 연관 관계 : 2개 이상의 사물이 연관 ( 실선 )
- 집합 관계 : 하나가 다른 것에 포함 ( 속이 빈 마름모 )
- 포함 관계 : 특수한 집합 관계; 영향을 미치는 관계 ( 채워진 마름모 )
- 일반화 관계 : 구체적인 개념과 일반적인 개념의 관계 ( 상위 쪽으로 속이 빈 화살표 )
- 의존 관계 : 연관은 있지만 always가 아닌 필요한 순간만 ( 점선 화살표 )
- 실체화 관계 : 할 수 있거나 해야하는 기능으로 그룹화 할 수 있는 관계; 연관 관계의 확장판? ( 빈 점선 화살표 )

## CASE ( Computer-aided software engineering )

- 시스템 개발 방법론들의 자동화를 지원하는 소프트웨어 도구를 제공해 개발자의 반복적인 작업량을 줄이도록 하느 것
- 소프트웨어 생명 주기 전 단계의 연결

## Use Case 구성 요소간의 관계

- 연관 관게 ( Association ) : 유스케이스와 액터 간의 상호작용이 있음을 표현
- 포함 관게 ( Include ) : 하나의 유스케이스가 다른 유스케이스의 실행을 전제로 할 때 형성되는 관계 
- 확장 관계 ( Extend ) : 확장 기능 유스케이스와 확장 대상 유스케이스 사이에 형성되는 관게
- 일반화 관계 ( Generalization ) : 유사한 유스케이스 또는 액터를 모아 추상화한 유스케이스 또는 액터와 연결시켜 그룹을 만들어 이해도를 높이기 위한 관계

`액터` : 시스템과 상호작용 하는 모든 것( 사람, 기계, 시스템)

## 객체 지향 설계 원칙

SOLID라고 한다.

**SRP, (Single Responsibility Principle)**
`단일 책임 원칙` : 객체는 단 하나의 책임만 가져야 함.

**OCP, (Open-Closed Principle)**
`개방-폐쇄 원칙` : 확장에 대해 열려 있어야 하고 변경에 대해 닫혀 있어야 함.

**LSP, (Liskov Sustiution Principle)**
`리스코프 치환 원칙` : 특정 메소드가 상위 타입을 인자로 사용할 때 하위 타입도 문제 없이 작동해야 함

**ISP, (Interface Segregation Principle)**
`인터페이스 분리 원칙` : 사용하지 않는 인터페이스는 영향 받지 않아야 함.

**DIP, (Dependency Inversion Principle)**
`의존 역전 원칙` : 상위 계층이 하위 계층에 의존하는 전통적인 의존관계를 역전 시킴으로써 상위 계층이 하위 계층의 구현으로부터 독립되게 할 수 있음

  - `기능성` (적합성, 정밀성, 상호 운용성, 보안성 준수성)
  - `신뢰성` (성숙성, 고장 허용성, 회복성)

  - `유지 보수성` (분석성, 변경성, 안정성, 시험성)

  - 기능 적합성, 성능 효율성, 호환성, 사용성, 신뢰성, 보안성, 유지 보수성, 이식성

## 소프트웨어 법칙

`Pareto의 법칙` : 오류의 80%는 전체의 20%내에서 발견된다는 법칙

`Brooks의 법칙` : 지연되는 프로젝트에 인력을 더 투입하면 오히려 더 늦어짐

`Pesticide Paradox` : 동일 테스트 케이스로 동일 테스트 반복 시 더 이상 결함이 발견되지 않는 현상

`부재의 궤변` : 결함을 모두 제거해도 사용자 요구사항을 만족시키지 못하면 해당 소프트웨어는 품질이 높다고 말할 수 없음
