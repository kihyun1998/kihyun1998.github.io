# 오답노트
---

## 요구사항 검토 방법

- `동료 검토` : 작성자가 명세서 내용 설명, 동료들이 결함 발견하는 형태

- `워크 스루` : 검토 회의 전 명세서를 미리 배포하여 사전 검토해서 짧은 검토 회의를 통해 오류 조기 검출

- `인스펙션` : 명세서 작성자를 제외한 다른 검토 전문가들이 확인하면서 결함 발견


---


## 기능적 요구 vs 비기능적 요구

- **기능적 요구사항** : 시스템이 실제 어떻게 동작하는지에 관점
ex) 시스템이 뭘 하는지, 어떤 기능?, 입출력 무엇, 어떤걸 저장, 반드시 수행해야하는 기능, 제공 받기 원하는 기능
- **비기능적 요구사항** : 시스템 구축에 대한 `성능`, `보안`, `품질`, `안정` 등 실제 수행에 보조적인 요구사항 ( like 속도 등..)
    **성능** : 처리 속도, 처리량 등 
    **보안** : 시스템의 데이터 및 기능, 운영 접근 통제
    **품질** : 품질 평가 대상에 대한

---


### UML 다이어그램 - 관계

- 연관 관계 : 2개 이상의 사물이 연관 ( 실선 )
- 집합 관계 : 하나가 다른 것에 포함 ( 속이 빈 마름모 )
- 포함 관계 : 특수한 집합 관계; 영향을 미치는 관계 ( 채워진 마름모 )
- 일반화 관계 : 구체적인 개념과 일반적인 개념의 관계 ( 상위 쪽으로 속이 빈 화살표 )
- 의존 관계 : 연관은 있지만 always가 아닌 필요한 순간만 ( 점선 화살표 )
- 실체화 관계 : 할 수 있거나 해야하는 기능으로 그룹화 할 수 있는 관계; 연관 관계의 확장판? ( 빈 점선 화살표 )

## 디자인 패턴

**생성 패턴**
`추상 팩토리(조립 공장)`, `빌더(건축가)`, `팩토리 메소드(완성품 공장)`, `프로토타입(복사)`, `싱글톤(정수기)`

**구조 패턴**
`어댑터(호환성 변압기)`, `브리지(분리 후 다리)`, `컴포지트(구분없이 합성)`, `데코레이터(눈사람 확장 장식)`, `퍼싸드(like 포장지)`, `플라이웨이트(공유해서 가볍게)`, `프록시(대리 수행)`

**행위 패턴**
`책임 연쇄(연속적으로 책임 분할)`, `커맨드(캡슐화 명령어)`, `인터프리터(언어 번역)`, `Iterator(같은 명령 반복)`, `Mediator(중계)`, `메멘토(롤백)`, `옵저버(감시 후 보고)`, `상태(상태마다 다른 처리)`, `Strategy(원하는 전략 사용)`,`템플릿 메소드(작은거 묶기)`, `방문자(기능 분리 후 방문)`

**추가**
`Factory Method ( 팩토리 메소드 )` : 서브 클래스에서 처리하도록 분리하여 캡슐화한 패턴

## 연계 시스템 구성

- `송신 시스템` : 데이터를 DB와 앱으로부터 연계 테이블 또는 파일 형태로 생성하여 송신
- `수신 시스템` : 수신한 연계 테이블, 파일 데이터를 관리하는 데이터 형식에 맞게 변환하여 DB에 저장 또는 앱에 제공
- `중계 서버` : 송 수신 시스템 사이에서 데이터 송수신하고 모니터링, 보안 강화 지원 등

## 객체지향 분석 방법론

- `Coad와 Yourdon` : E-R 다이어그램 사용, 데이터 모델링에 초점
- `Rumbaugh` : 가장 일반적, 객체/동적/기능 모델로 나눠 수행
- `Booch` : 미시적 (Micro) 개발 프로세스와 거시적 (Marco) 개발 프로세스 모두 사용
- `Jacobson` : Use Case를 강조하여 사용
- `Wirfs-Brock` : 분석과 설계 구분 없음, 고객 명세서를 평가해서 연속적으로 수행하는 기법

## 화면 설계

**ISO/IEC 9126**
  - 기능성, 신뢰성, 사용성, 효율성, 유지 보수성, 이식성
  - `기능성` (적합성, 정밀성, 상호 운용성, 보안성 준수성)
  - `신뢰성` (성숙성, 고장 허용성, 회복성)
  - `사용성` (이해성, 학습성, 운용성, 친밀성)
  - `효율성` (시간, 자원 효율성)
  - `유지 보수성` (분석성, 변경성, 안정성, 시험성)
  - `이식성` (적용성, 설치성, 대체성, 공존성)

**ISO/IEC 25010**
  - 기능 적합성, 성능 효율성, 호환성, 사용성, 신뢰성, 보안성, 유지 보수성, 이식성

**ISO/IEC 12119**
  - 테스트 절차 포함

**ISO/IEC 14598**
  - 각 활동 별 제품 평가 활동

## 소프트웨어 법칙

`Pareto의 법칙` : 오류의 80%는 전체의 20%내에서 발견된다는 법칙

`Brooks의 법칙` : 지연되는 프로젝트에 인력을 더 투입하면 오히려 더 늦어짐

`Pesticide Paradox` : 동일 테스트 케이스로 동일 테스트 반복 시 더 이상 결함이 발견되지 않는 현상

`부재의 궤변` : 결함을 모두 제거해도 사용자 요구사항을 만족시키지 못하면 해당 소프트웨어는 품질이 높다고 말할 수 없음
