## 요구사항 검토 방법

- `동료 검토` : 작성자가 명세서 내용 설명, 동료들이 결함 발견하는 형태

- `워크 스루` : 검토 회의 전 명세서를 미리 배포하여 사전 검토해서 짧은 검토 회의를 통해 오류 조기 검출

- `인스펙션` : 명세서 작성자를 제외한 다른 검토 전문가들이 확인하면서 결함 발견

## 기능적 요구 vs 비기능적 요구

- **기능적 요구사항** : 
- 시스템이 실제 어떻게 동작하는지에 관점
- 시스템이 수행해야 하는 행위를 구체화
- 제공해야할 기능 정의
- 입출력, 데이터베이스, 통신 등

ex) 시스템이 뭘 하는지, 어떤 기능?, 입출력 무엇, 어떤걸 저장, 반드시 수행해야하는 기능, 제공 받기 원하는 기능

- **비기능적 요구사항** : 
- 시스템 구축에 대한 `성능`, `보안`, `품질`, `안정` 등 실제 수행에 보조적인 요구사항 ( like 속도 등..)
    **성능** : 처리 속도, 처리량 등 
    **보안** : 시스템의 데이터 및 기능, 운영 접근 통제
    **품질** : 품질 평가 대상에 대한
- 품질, 제약사항
- 사용 용이성, 효율성, 신뢰성, 이식성, 유연성, 확장성
- **성능** : 응답 속도, 자원 사용량
- **보안** : 침입 대응, 침입 탐지, 사용자 인증, 구너한 부여

### UML 다이어그램 - 관계

- 연관 관계 : 2개 이상의 사물이 연관 ( 실선 )
- 집합 관계 : 하나가 다른 것에 포함 ( 속이 빈 마름모 )
- 포함 관계 : 특수한 집합 관계; 영향을 미치는 관계 ( 채워진 마름모 )
- 일반화 관계 : 구체적인 개념과 일반적인 개념의 관계 ( 상위 쪽으로 속이 빈 화살표 )
- 의존 관계 : 연관은 있지만 always가 아닌 필요한 순간만 ( 점선 화살표 )
- 실체화 관계 : 할 수 있거나 해야하는 기능으로 그룹화 할 수 있는 관계; 연관 관계의 확장판? ( 빈 점선 화살표 )

## UML 의미적 관계

-` Dependency ( 의존 )`: 한 사물의 명세서가 바뀌면 그것을 사용하는 다른 사물에게 **영향을 끼치는** 것

- `Realization ( 실체화 )` : 한 객체가 다른 객체에 의해 오퍼레이션을 **수행하도록 지정**

- `Generalization ( 일반화 )` : 일반화된 사물과 좀 더 특별한 사물 사이의 관계

- `Association ( 연관 )` : 두 사물관 구조적 관계, 어느한 사물 객체가 다른 사물 객체와 연결되어 있음

## Use Case 구성 요소간의 관계

- 연관 관게 ( Association ) : 유스케이스와 액터 간의 상호작용이 있음을 표현
- 포함 관게 ( Include ) : 하나의 유스케이스가 다른 유스케이스의 실행을 전제로 할 때 형성되는 관계 
- 확장 관계 ( Extend ) : 확장 기능 유스케이스와 확장 대상 유스케이스 사이에 형성되는 관게
- 일반화 관계 ( Generalization ) : 유사한 유스케이스 또는 액터를 모아 추상화한 유스케이스 또는 액터와 연결시켜 그룹을 만들어 이해도를 높이기 위한 관계

`액터` : 시스템과 상호작용 하는 모든 것( 사람, 기계, 시스템)

## 객체 지향 설계 원칙

SOLID라고 한다.

**SRP, (Single Responsibility Principle)**
`단일 책임 원칙` : 객체는 단 하나의 책임만 가져야 함.

**OCP, (Open-Closed Principle)**
`개방-폐쇄 원칙` : 확장에 대해 열려 있어야 하고 변경에 대해 닫혀 있어야 함.

**LSP, (Liskov Sustiution Principle)**
`리스코프 치환 원칙` : 특정 메소드가 상위 타입을 인자로 사용할 때 하위 타입도 문제 없이 작동해야 함

**ISP, (Interface Segregation Principle)**
`인터페이스 분리 원칙` : 사용하지 않는 인터페이스는 영향 받지 않아야 함.

**DIP, (Dependency Inversion Principle)**
`의존 역전 원칙` : 상위 계층이 하위 계층에 의존하는 전통적인 의존관계를 역전 시킴으로써 상위 계층이 하위 계층의 구현으로부터 독립되게 할 수 있음

## 소프트웨어 법칙

`Pareto의 법칙` : 오류의 80%는 전체의 20%내에서 발견된다는 법칙

`Brooks의 법칙` : 지연되는 프로젝트에 인력을 더 투입하면 오히려 더 늦어짐

`Pesticide Paradox` : 동일 테스트 케이스로 동일 테스트 반복 시 더 이상 결함이 발견되지 않는 현상

`부재의 궤변` : 결함을 모두 제거해도 사용자 요구사항을 만족시키지 못하면 해당 소프트웨어는 품질이 높다고 말할 수 없음

## 주요 아키텍처 패턴

`레이어 패턴 ( Layer Pattern )` : 
  - 시스템을 계층으로 구분
  - 고전적 방법
  - 서브 시스템들이 계층 구조를 이루고 하위 계층은 상위 계층에 대한 서비스 제공자가 되고 상위 계층은 하위 계층의 클라이언트가 된다.

`클라이언트-서버 패턴` :
  - 하나의 서버 컴포넌트와 다수의 클라이언트 컴포넌트로 구성되는 패턴
  - 클라이언트가 서버에 요청하고 응답을 받아 사용자에게 제공

`파이프-필터 패턴` :
  - 데이터 스트림 절차의 각 단계를 필터 컴포넌트로 캡슐화하여 파이프를 통해 데이터를 전송하는 패턴

`모델-뷰-컨트롤러 패턴` :
  - 서브시스템을 모델, 뷰, 컨트롤러의 세 부분으로 구조화하는 패턴


---


## 시퀀스 다이어그램의 구성 요소

`액터` (Actor) : 시스템으로부터 서비스를 요청하는 외부 요소로 사람이나 외부 시스템을 의미
`객체`(Object) : 메시지를 주고받는 주체
`라이프라인` (Lifeline) : 객체가 메모리에 존재하는 기간
`활성 상자` (Activation Box) : 객체가 메시지를 주고 받으며 구동되고 있음을 표현
`메시지` (Message) : 객체가 상호 작용을 위해 주고받는 메시지
`객체 소멸` : 라이프라인 상에서 객체 소멸 표시를 만나면 해당 객체는 더 이상 메모리에 존재하지 않음을 의미
`프레임` (Frame) : 다이어그램의 전체 또는 일부를 묶어 표현


---


## MVC 패턴

서브 시스템을 Model, View, Controller 세 부분으로 구조화하는 패턴

- `모델` : 서브시스템의 핵심 기능과 데이터를 보관
- `뷰` : 사용자에게 정보 표시
- `컨트롤러` : 사용자로부터 받은 입력을 처리하며 처리 결과에 따라 모델의 상태를 갱신함

- 각 부분은 별도의 컴포넌트로 분리되어 있어서 서로 영향을 받지 않고 작업할 수 있음
- 여러 개의 뷰를 만들 수 있어서 한 개의 모델에 대한 여러 개의 뷰를 필요로 하는 대화형 어플리케이션 적합


---


## 다형성 (Polymorphism)

- 메시지에 의해 객체가 연산을 수행하게 될 때 하나의 메시지에 대해 각각의 객체가 가지고 있는 고유한 방법으로 **응답할 수 있는 능력**을 의미함
- 객체들은 동일한 메소드명을 사용하며 같은 의미의 응답을 함
- 응용 프로그램 상에서 하나의 함수나 연산자가 두 개 이상의 **서로 다른 클래스의 인스턴스들을 같은 클래스에 속한 인스턴스처럼 수행할 수 있도록 하는 것**
- 오버 로딩 : 한 클래스 내에서 메서드의 이름은 동일하지만 매개변수의 수나 타입을 다르게 하여 재정의 하는 것
- 오버라이딩 : 상속 관계에서만 발생하는데 슈퍼클래스의 메서드를 서브클래스에서도 동일한 메스드를 재정의 하는 것

## 용어

`EAI` ( Enterprise Application Integration ) :
  - 기업 응용 프로그램 통합으로 기업용 응용 프로그램의 구조적 통합 방안을 가리킴

`FEP` ( Front-End-Processor ) :
  - 입력되는 데이터를 컴퓨터의 프로세서가 처리하기 전에 미리 처리하여 프로세서가 차지하는 시간을 줄여주는 프로그램이나 하드웨어

  - 자유 소프트웨어 재단에서 만든 자유 소프트웨어 라이선스

`Duplexing` :
  - 이중화 ( 데이터베이스의 회복 기법 중 가장 간단한 것 )
