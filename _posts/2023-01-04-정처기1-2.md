# **Section 02.** 스크럼(Scrum) 기법
---

## 스크럼이란?

스크럼에는 팀이 중심이 되어 개발의 효율성을 높인다는 의미가 담겨있습니다.


## 구성원
- 제품 책임자 ( PO : Product Owner )
- 스크럼 마스터 ( SM : Sccrum Master )
- 개발팀 ( DT : Development Team )


## 역할

### 제품 책임자

이해관계자들 중 개발될 제품에 대한 이해도가 높고 요구사항을 책임지고 의사 결정할 사람을 선정
주로 개발 의뢰자나 사용자가 담당함.

- 제품에 대한 요구사항을 작성하는 주체
- 요구사항이 담긴 백로그를 작성 및 백로그 우선순위 지정
- 테스트를 진행하면서 주기적으로 요구사항 우선순위 갱신

### 스크럼 마스터

스크럼 팀에게 객관적 조언을 해주는 가이드 역할, 팀원들을 통제하는 것이 목표가 아님.

- 진행사항 점검
- 장애 요소를 공론화하여 처리

### 개발팀

제품 책임자와 스크럼 마스터를 제외한 모든 팀원, 디자이너, 테스터 등등
인원은 7~8명이 적당하다.


## 스크럼 개발 과정

### 제품 백로그

제품 개발에 필요한 모든 요구사항(User Story)을 우선순위에 따라 나열한 목록

- 지속적으로 업데이트 됨.
- 요구사항을 기반으로 전체 일정 계획(Release Plan)을 수립함.

### 스프린트 계획 회의(Sprint Planning Meeting)

제품 백로그 중 이번 스프린트에서 수행할 작업을 대상으로 단기 일정 수립

- Task라는 작업 단위로 분할하여 스프린트 백로그를 작성

### 스프린트(Sprint)

실제 개발 작업을 진행하는 과정 ( 2 ~ 4주 소요 )
스프린트 백로그에 작성된 태스크를 대상으로 `속도(Velocity)` 추정 후 개발 담당자에게 할당

`속도(Velocity)` : 한 번의 스프린트에서 한 팀이 감당할 수 있는 제품 백로그 양 ( 추정치 )

- 태스크 할당할 때는 개발자가 원하는 태스크를 할당하는 것이 좋음.
- 개발 담당자에게 할당한 태스크는 할 일( To Do ), 진행 중( In Progress ), 완료( Done )의 상태를 갖음.

### 일일 스크럼 회의(Daily Scrum Meeting)

모든 팀원이 매일 약속된 시간에 짧게 진행 상황 점검 ( 15분 정도 )

- 보통 서서 진행
- 남은 작업 시간은 소멸 차트( Burn-down Chart )에 표시
- 스크럼 마스터는 장애 요소 해결을 도와줌.

### 스프린트 검토 회의(Sprint Review)

요구사항에 잘 부합되는지 사용자가 포함된 참석자 앞에서 테스팅

- 한 주당 한 시간 내에서 진행
- 제품 책임자는 개선할 사항에 대한 피드백 정리 후 제품 백로그 업데이트

### 스프린트 회고(Sprint Retrospective)

스프린트 주기를 되돌아보며 규칙을 잘 준수 했는지, 개선할 점 등 기록하며 회고
시점은 스프린트가 끝난 시점 또는 일정 주기로 수행
