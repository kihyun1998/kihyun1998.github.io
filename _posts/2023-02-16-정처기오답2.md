# 오답노트
---

## 화이트박스 테스트

- 설계 절차에 초점
- 원시코드 오픈
- 한 문장씩 수행
- 선택, 반복 등 분기점 부분들 수행 > 논리적 경로 제어
- **종류** : `기초 경로 검사`, `제어 구조 검사`(`조건 검사`, `루프 검사`, `데이터 흐름 검사`)


---


## 블랙박스 테스트 

- 구조를 고려하지 않고 요구나 명세를 기초로 결정
- 인터페이스에서 실시
- 누락된 기능, 오류 발견을 위해 사용
- **종류** : `동치 분할 검사`, `경계값 분석`, `원인-효과 그래프 검사`, `오류 예측 검사`, `비교 검사`


---


## 인터페이스 보안 기능

- 네트워크 영역 : 트래픽에 대한 암호화 설정
    - IPSec, SSL, S-HTTP
- 애플리케이션 영역 : 코드 상 보안 취약점 보완
- 데이터베이스 영역 : DB 동작 객체의 보안 취약점 보완
    - 민감 데이터 암호화, 익명화


---


## 인퍼테이스 구현 검증 도구

`xUnit` : 다양한 언어에 적용되는 단위 테스트 프레임워크
`STAF` : 서비스 호출 및 컴포넌트 재사용 등 다양한 환경을 지원
`FitNesse` : 웹 기반 테스트케이스 설계, 실행 결과 확인 등
`NTAF` : FitNesse와 STAF의 장점 통합
`watir` : Ruby를 사용하는 애플리케이션 전용


---


## FTR ( 정형 기술 검토 )

- 오류 검출에 초점을 두고 해결책 나중에 미룸 ( 제품 검토의 집중성 )
- 검토를 위한 자료를 사전에 배포하여 검토 ( 사전 준비성 )
- 의견을 제한하되 충분히 받아들임 ( 의제의 제한성 )
- 안건을 세우며 고수 ( 안건 고수성 )
- 논쟁과 반박을 제한 ( 논쟁 반박의 제한성 )
- 문제 영역을 공개 ( 문제 공개성 )
- 참가자 수 제한 ( 참가 인원의 제한성 )
- 발견된 오류는 문서화 ( 문서성 )

## 테스트

`테스트 케이스` : 사용자 요구사항을 준수했는지 확인하기 위해 설계된 입력 값, 설계 조건, 기대 결과 등에 대한 명세서

`테스트 시나리오` : 테스트 케이스를 적용하는 순서에 따라 여러 개의 테스트 케이스들을 묶은 집합

`테스트 오라클` : 테스트 결과가 올바른지 판단하기 위해 사전 정의된 참 값을 대입하여 비교하는 기법
    - **종류** : 참 오라클, 샘플링 오라클, 추정 오라클, 일관성 오라클

## 개발 단계에 따른 애플리케이션 테스트

- `단위 테스트`
  - 최소 단위에 집중해서 테스트

- `통합 테스트`
  - 단위 테스트 후 모듈 결합하는 과정 테스트

- `시스템 테스트`
  - 소프트웨어가 컴퓨터 시스템에 완벽 수행되는지 테스트 (기능적, 비기능적 요구사항)

- `인수 테스트`
  - 사용자의 요구사항을 충족하는지 테스트
  - 알파 테스트, 베타 테스트
  - 사용자 인수 테스트, 운영상의 인수 테스트, 계약 인수 테스트, 규정 인수 테스트

- 텍스트 드라이버
  - 상향식 통합 테스트
  - 하위 모듈 호출

- 테스트 스텁
  - 하향식 통합 테스트
  - 시험용 모듈

## 소프트웨어 테스트 검증과 확인

`검증 ( Verification )` : 소프트웨어가 **요구사항에 부합**하게 구현됐음을 보장하는 활동 (개발자)

`확인 ( Validation )` : 소프트웨어가 **고객의 의도에 따라 구현**됐음을 보장하는 활동 (사용자)

## NS Chart ( Nassi-Schneiderman )

- 3가지 기본구조만으로 논리를 표현(표준화 기능)
- 논리 기술에 중점을 둔 도형을 이용한 표현 방법
- 순차, 반복, 선택, 다중 선택 구조 등 표현
- GOTO나 화살표 사용하지 않음, 선택과 반복 구조를 시각화
- 조건이 복합되어 있는 곳의 처리를 시작적으로 명확히 식별
- 기본구조의 입구와 출구는 각 하나씩
- 전체적인 알고리즘을 일목요연하게 볼 수 있음

## 정렬

`퀵 정렬` : 레코드의 많은 자료 이동을 없애고 하나의 파일을 부분적으로 나눠가며 정렬하는 방식

`삽입 정렬` : 모든 요소를 정렬된 부분과 비교하여 정렬하는 방식

`쉘 정렬` : 삽입 정렬의 확장 개념, 입력 파일을 매게변수값으로 서브파일을 구성하고 서브파일을 삽입정렬 방식으로 순서 배열하는 과정 반복

`선택 정렬` : 최솟값 찾아서 최솟값 위치에 넣는 정렬

`버블 정렬` : 인접한 값 비교하여 교환하는 정렬

`힙 정렬` : 전 이진 트리를 이용한 정렬 방식

`2-Way 합병 정렬` : 이미 정렬된 두 파일을 한 개의 파일로 합병하는 정렬 방식

`기수 정렬` : 키 값을 분석해서 같은 값끼리 순서에 맞게 버킷에 분배하고 버킷의 순서대로 레코드 꺼내 정렬


---


## 검색

`선형 검색` : 처음부터 끝까지 하나씩 순서대로 비교하며 원하는 값을 찾아내는 검색
`이진 검색` : 검색을 수행하기 전에 반드시 데이터의 집합이 정렬되어 있어야 함.

## 소프트웨어 버전 관리 도구

`RCS ( Revision Control System )`
- CVS와 차이점은 소스 파일의 수정을 한 사람만으로 제한
- 다수의 사용자가 동시에 파일 수정을 할 수 없도록 파일 잠금 방식으로 버전 관리
- 다른 방향으로 진행된 개발 결과를 합치거나 변경 내용을 추적

`RPC ( Remote Procedure Call )`
- 별도의 원격 제어를 위한 코딩 없이 다른 주소 공간에서 리모트의 함수나 프로시저를 실행할 수 있게 해주는 프로세스간 통신


---


## EAI ( Enterprise Application Integration )

기업 내 각종 애플리케이션 및 플랫폼 간의 정보 전달 연계 통합 등 상호연동이 가능하게 해주는 솔루션

- `Point-to-Point` : 가장 기본적인 애플리케이션 통합 방식 1:1로 연결

- `Hub & Spoke` : 단일 접점인 허브 시스템을 통해 데이터 전송하는 중앙 집중형 방식

- `Message Bus` : 애플리케이션 사이에 미들웨어를 두어 처리하는 방식

- `Hybrid` : Hub & Spoke와 Message Bus 혼합 방식


---


## 소프트웨어 재공학

`Analysis` : 기존 소프트웨어를 **분석**하여 재공학 대상을 **선정**

`Migration` : 기존 소프트웨어를 ***다른 운영체제***나 **하드웨어 환경**에서 사용할 수 있도록 **변환**하는 작업

`Restructuring` : 기존 소프트웨어를 향상시키기 위해 **코드를 재구성**하는 작업

`Reverse Engineering` : 기존 소프트웨어를 분석하여 **소스코드를 얻어**내는 작업
