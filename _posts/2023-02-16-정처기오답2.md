# 오답노트
---

## 화이트박스 테스트

- 설계 절차에 초점
- 원시코드 오픈
- 한 문장씩 수행
- 선택, 반복 등 분기점 부분들 수행 > 논리적 경로 제어
- **종류** : `기초 경로 검사`, `제어 구조 검사`(`조건 검사`, `루프 검사`, `데이터 흐름 검사`)


---


## 블랙박스 테스트 

- 구조를 고려하지 않고 요구나 명세를 기초로 결정
- 인터페이스에서 실시
- 누락된 기능, 오류 발견을 위해 사용
- **종류** : `동치 분할 검사`, `경계값 분석`, `원인-효과 그래프 검사`, `오류 예측 검사`, `비교 검사`


---


## 인터페이스 보안 기능

- 네트워크 영역 : 트래픽에 대한 암호화 설정
    - IPSec, SSL, S-HTTP
- 애플리케이션 영역 : 코드 상 보안 취약점 보완
- 데이터베이스 영역 : DB 동작 객체의 보안 취약점 보완
    - 민감 데이터 암호화, 익명화


---


## 인퍼테이스 구현 검증 도구

`xUnit` : 다양한 언어에 적용되는 단위 테스트 프레임워크
`STAF` : 서비스 호출 및 컴포넌트 재사용 등 다양한 환경을 지원
`FitNesse` : 웹 기반 테스트케이스 설계, 실행 결과 확인 등
`NTAF` : FitNesse와 STAF의 장점 통합
`watir` : Ruby를 사용하는 애플리케이션 전용


---


## FTR ( 정형 기술 검토 )

- 오류 검출에 초점을 두고 해결책 나중에 미룸 ( 제품 검토의 집중성 )
- 검토를 위한 자료를 사전에 배포하여 검토 ( 사전 준비성 )
- 의견을 제한하되 충분히 받아들임 ( 의제의 제한성 )
- 안건을 세우며 고수 ( 안건 고수성 )
- 논쟁과 반박을 제한 ( 논쟁 반박의 제한성 )
- 문제 영역을 공개 ( 문제 공개성 )
- 참가자 수 제한 ( 참가 인원의 제한성 )
- 발견된 오류는 문서화 ( 문서성 )


---



## 테스트

`테스트 케이스` : 사용자 요구사항을 준수했는지 확인하기 위해 설계된 입력 값, 설계 조건, 기대 결과 등에 대한 명세서

`테스트 시나리오` : 테스트 케이스를 적용하는 순서에 따라 여러 개의 테스트 케이스들을 묶은 집합

`테스트 오라클` : 테스트 결과가 올바른지 판단하기 위해 사전 정의된 참 값을 대입하여 비교하는 기법
    - **종류** : 참 오라클, 샘플링 오라클, 추정 오라클, 일관성 오라클 


---


## 개발 단계에 따른 애플리케이션 테스트

- `단위 테스트`
  - 최소 단위에 집중해서 테스트

- `통합 테스트`
  - 단위 테스트 후 모듈 결합하는 과정 테스트

- `시스템 테스트`
  - 소프트웨어가 컴퓨터 시스템에 완벽 수행되는지 테스트 (기능적, 비기능적 요구사항)

- `인수 테스트`
  - 사용자의 요구사항을 충족하는지 테스트
  - 알파 테스트, 베타 테스트
  - 사용자 인수 테스트, 운영상의 인수 테스트, 계약 인수 테스트, 규정 인수 테스트
