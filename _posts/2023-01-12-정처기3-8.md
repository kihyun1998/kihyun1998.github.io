# **Section 28.** 디자인 패턴
---


## 디자인 패턴이란

세부적인 구현 방안을 설계할 때 참조할 수 있는 전형적인 해결방식, 예제

### 특징

- 문제 및 배경, 실제 적용된 사례, 재사용이 가능한 샘플 코드로 구성
- "바퀴를 다시 발명하지 마라"는 문구 처럼 문제가 발생하면 새 해결책 보다는 문제에 해당하는 디자인 패턴 참조
- 디자인 패턴은 1995년 GoF가 구체화 체계화 함 > 가장 많이 사용되는 디자인 패턴

### 장점

- 범용적 코딩스타일로 구조 파악 용이
- 객체 지향 설계 및 구현의 생산성 높이는데 적합
- 재사용을 통해 개발 시간 및 비용 절약
- 원활한 의사소통 가능
- 설계 변경에 대한 유연한 대처 가능

### 단점

- 초기 투자 비용 부담
- 객체 지향 설계만 다루기 때문에 다른 기반 개발에 적합하지 않음


### 생성 패턴 ( Creational Pattern )

객체의 생성과 참조 과정을 캡슐화 하여 객체가 생성, 변경되어도 프로그램 구조에 영향을 받지 않도록 유연성을 더해줌


- **추상 팩토리 ( Abstract Factory )**
    - 클래스에 의존하지 않고 인터페이스를 통해 서로 연관 의존 객체들의 그룹을 추상적으로 표현
    - 연관된 서브 클래스를 묶어 한 번에 교체하는 것 가능
    - 서로 다른 부품만 조립하는 `조립 공장`
- **빌더 ( Builder )**
    - 인스턴스를 건축하듯 조합하여 객체 생성
    - 생성 과정과 표현 방법을 분리하여 동일 객체 생성이여도 다른 결과를 만들 수 있음
    - `건축가`
- **팩토리 메소드 ( Factory Method )**
    - 객체 생성을 서브 클래스에서 처리하도록 분리하여 캡슐화된 패턴
    - 상위 클래스에서 인터페이스 정의, 서브 클래스에서 실제 생성
    - 가상 생성자(Virtual Constructor) 패턴이라고도 함
    - 부품부터 완성품까지 `통째로 찍어내는 공장`
- **프로토타입 ( Prototype )**
    - 원본 객체를 복제하여 객체 생성
    - 비용이 큰 경우 사용
    - `원형`을 두고 복제품을 만드는 것
- **싱글톤 ( Singleton )**
    - 생성된 객체는 어디서든 참조할 수 있지만 여러 프로세스가 동시에 참조할 순 없음
    - 인스턴스가 하나뿐임을 보장하고 불필요한 메모리 낭비를 최소화 함.
    - 식당에서 누구나 사용할 수 있는 `하나뿐인` 정수기


### 구조 패턴 ( Structural Pattern )

클래스나 객체들을 조합하여 더 큰 구조를 만들 수 있게 해주는 패턴

- **어댑터 ( Adapter )**
    - 호환성이 없는 클래스들의 인터페이스를 호환 가능하도록 변환해주는 패턴
    - 인터페이스가 일치하지 않을 때 이용
    - 전압을 맞춰주는 `변압기`
- **브리지 ( Bridge )**
    - 구현분에서 추상층을 분리, 서로 독립적으로 확장할 수 있도록 구성
    - 기능과 구현을 두 개의 별도 클래스로 구현
    - 두 섬을 연결하는 `다리`
- **컴포지트 ( Composite )**
    - 복합 객체와 단일 객체를 구분 없이 다루고자 할 때 사용하는 패턴
    - 객체를 트리 구조로 구성 
    - 폴더와 파일을 `합성`한 것
- **데코레이터 ( Decorator )**
    - 객체간 결합을 통해 능동적으로 기능을 확장할 수 있는 패턴
    - 객체에 부가적인 기능 추가를 위해 다른 객체를 덧붙이는 방식으로 구현
    - 온갖 것으로 `장식된` 눈사람
- **퍼싸드 ( Facade )**
    - 복잡한 서브 클래스들을 피해 더 상위에 인터페이스를 구현하여 서브 클래스 기능 간단하게 사용할 수 있도록 하는 패턴
    - 서브 클래스들 사이의 통합 인터페이스를 제공하는 Wrapper 객체가 필요
    - `외부`의 리모콘 버튼만으로 복잡한 명령 간편히 수행
- **플라이웨이트 ( Flyweight )**
    - 인스턴스가 필요할 때 매번 생성하는 것이 아닌 공유해서 사용해서 메모리 절약하는 패턴
    - 다수의 유사 객체 생성, 조작 때 유용
    - 부담을 `가볍게`하기 위해 물품을 공유
- **프록시 ( Proxy )**
    - 접근 어려운 객체 사이에 인터페이스 역할을 수행하는 패턴
    - 네트워크 연결, 메모리의 대용량 객체로 접근에 이용
    - 법률 업무를 `대리`해서 처리해주는 변호사


### 행위 패턴 ( Behavioral Pattern )

서로 상호작용 하는 방법이나 책임 분배 방법을 정의하는 패턴
하나의 객체로 수행할 수 없는 작업을 여러 객체로 분해하면서 결합도를 최소화 하도록 도움

- **책임 연쇄 ( Chain of Responsibility )**
    - 요청 처리 객체가 둘 이상 존재해서 한 객체가 처리 못하면 다음 객체로 넘어감
    - 객체들이 고리로 묶여서 요청이 처리될 때 까지 책임이 넘어감
    - 위에서 쏟아지는 물을 여러 물받이가 `연속`해서 `나눠 받는` 물레방아
- **커맨드 ( Command )**
    - 요청을 객체형태로 캡슐화하여 재이용하거나 취소할 수 있도록 로그 남기는 패턴
    - 요청에 사용되는 명령어들을 추상 클래스와 구체 클래스로 분리하여 단순화
    - `명령어`를 하나로 합쳐둔 것
- **인터프리터 ( Interpreter )**
    - 언어에 문법 표현을 정의하는 패턴
    - SQL, 통신 프로토콜 개발에 사용
    - `언어 번역가`
- **반복자 ( Iterator )**
    - 자료구조같은 접근이 잦은 객체에 동일한 인터페이스를 사용하도록 하는 패턴
    - 내부 표현 방법의 노출 없이 순차적 접근 가능
    - 음악 파일의 다음 곡 재생처럼 `같은 명령의 반복`
- **중재자 ( Mediator )**
    - 상호작용을 캡슐화하여 객체로 정의하는 패턴
    - 객체 사이의 의존성을 줄여 결합도 감소
    - 객체 간의 통제와 지시 역할을 수행
    - 물품 매매를 `중개해주는` 인터넷 쇼핑몰
- **메멘토 ( Memento )**
    - 특정 시점 객체 내부 상태 객체화하여 되돌릴 수 있는 기능 제공하는 패턴
    - ctrl + z 같은 되돌리기 기능을 개발할 때 이용
    - 기억 속의 `그 때로` 돌아가는 것
- **옵저버 ( Observer )**
    - 객체의 상태가 변화하면 객체에 상속되어 있는 다른 객체들에게 변화된 상태 전달
    - 분산된 시스템 간에 이벤트 생성, 발행 하고 이를 수신해야할 때 이용
    - 변화를 `지켜보고` 알려주는 것
- **상태 ( State )**
    - 상태에 따라 동일한 동작을 다르게 처리해야할 때 사용
    - 상태를 캡슐화하고 이를 참조하는 방식
    - 환자의 `상태`에 따라 치료 방법이 다른 것
- **전략 ( Strategy )**
    - 동일 계열의 알고리즘을 개별 캡슐화하여 상호 교환할 수 있게 정의하는 패턴
    - 클라이언트는 독립적으로 원하는 알고리즘 선택해서 사용, 클라이언트에 영향 없이 알고리즘 변경 가능
    - 여러 전력을 정하고 필요할 때 `원하는 전략`을 선택하여 쓰는 것
- **템플릿 메소드 ( Template Method )**
    - 상위 클래스에서 골격 정의, 하위 클래스에서 세부 처리를 구체화하는 구조의 패턴
    - 유사한 서브 클래스의 공통된 내용을 상위 클래스에 정의 > 코드 양 감소, 유지보수 용이
    - 세모, 네모, 동그라미를 그리는 `방법`들을 도형이라는 `하나의 큰 틀`로 묶는 것
- **방문자( Visitor )**
    - 클래스의 데이터 구조에서 처리 기능을 분리하여 별도의 클래스로 구성하는 패턴
    - 분리된 처리 기능은 각 클래스를 방문하여 수행
    - 책을 만들기 위해 저자, 편집자, 홍보팀을 번갈아가며 `방문`하는 것과 비교
