# **Section 57.** 개발 단계에 따른 애플리케이션 테스트
---

## 개발 단계에 따른 애플리케이션 테스트

개발단계에 따라 단위테스트, 통합 테스트, 시스템 테스트, 인수 테스트로 분류됨, 이를 테스트 레벨이라고 한다.

- 애플리케이션 테스트는 개발 단계에서부터 테스트를 수행하므로 코드 오류뿐만 아니라 요구 분석의 오류, 설계 인터페이스 오류 등을 발견
- 애플리케이션 테스트와 소프트웨어 개발 단계를 연결하여 표현한 것이 V-모델

  **(소프트웨어 개발 단계)**
    요구사항 (Requirements) > 분석 (Specification) > 설계 (Design) > 구현 (Code) >
  
  **(테스트 단계)**
    단위 테스트 (Unit Testing) > 통합 테스트 (Integration Testing) > 시스템 테스트 (System Testing) > 인수 테스트 (Acceptance Testing)

> 검증 (Verification) 테스트  : 개발자 기준의 테스트 (단위 테스트, 통합 테스트, 시스템 테스트)
> 확인 (Validation) 테스트 : 사용자 기준의 테스트 (인수 테스트)

### 단위 테스트

설계의 최소 단위인 `모듈`이나 `컴포넌트`에 초점을 맞춰 테스트하는 것

- 인터페이스, 외부적 I/O, 자료 구조 독립적 기초 경로, 오류 처리 경로, 경계 조건 등을 검사
- 사용자의 요구사항을 기반으로 한 가능성 테스트를 최우선 수행
- 구조 기반 테스트와 명세 기반 테스트로 나뉨 (주로 구조 기반 테스트로 함)
- **발견 가능한 오류** : 
    알고리즘 오류에 따른 원치 않는 결과, 탈출구가 없는 반복문, 틀린 계산 수식에 의한 잘못된 결과

|테스트 방법|테스트 내용|테스트 목적|
|---|-----|---|
|**구조 기반 테스트**|내부 구조 및 복잡도를 검증하는 화이트박스 테스트 시행|제어 흐름, 조건 결정|
|**명세 기반 테스트**|목적 및 실행 코드 기반의 블랙박스 테스트 시행|동긍 분할, 경계 값 분석|


### 통합 테스트

단위 테스트가 완료된 모듈들을 `결합`하여 하나의 시스템으로 완성시키는 과정에서의 테스트

- 모듈 간 또는 통합된 컴포넌트 간의 상호 작용 오류를 검사


### 시스템 테스트

소프트웨어가 컴퓨터 시스템에서 완벽하게 수행되는가 점검하는 테스트

- 환경적 장애 리스크 최소화하기 위해 실제 사용 환경과 유사한 테스트 환경에서 테스트 진행
- 기능적 요구사항과 비기능적 요구사항으로 구분하여 만족하는지 테스트

|테스트 방법|테스트 내용|
|--|------|
|**기능적 요구사항**|요구사항 명세서, 비지니스 절차, 유스케이스 등 명세서 기반의 블랙박스 테스트 시행|
|**비기능적 요구사항**|성능, 보안, 회복 테스트, 웹 네비게이션 등 구조적 요소에 대한 화이트박스 테스트 시행|


### 인수 테스트

사용자의 요구사항을 충족하는지에 중점을 두고 테스트하는 방법

- 개발한 소프트웨어를 사용자가 직접 테스트함.
- 문제가 없으면 사용자가 소프트웨어를 인수 & 프로젝트 종료

- 종류
    - **사용자 인수 테스트** : 
        - 사용자가 시스템 사용의 적절성 여부 확인
    - **운영상의 인수 테스트** : 
        - 시스템 관리자가 시스템 인수 시 수행하는 테스트 기법
        - 백업/복원 시스템, 재난 복구, 사용자 관리, 정기 정검 등을 확인 함.
    - **계약 인수 테스트** :
        - 계약상의 인수/검수 조건을 준수하는지 여부 확인
    - **규정 인수 테스트** :
        - 소프트웨어가 정부 지침, 법규 규정 등 규정에 맞게 개발되었는지 확인
    - **알파 테스트** : 
        - 개발자의 장소에서 사용자가 개발자 앞에서 행하는 테스트
        - 통제된 환경에서 수행
        - 오류와 사용상 문제점을 함께 확인하면서 기록
    - **베타 테스트** :
        - 선정된 사용자 앞에서 행하는 테스트 기법
        - 필드 테스트 (Field Testing)이라고 불림
        - 개발자에 의해 제어되지 않은 상태에서 사용자가 직접 테스트 하는 것
        - 발견된 오류와 사용상의 문제점 기록 후 개발자에게 주기적으로 보고
